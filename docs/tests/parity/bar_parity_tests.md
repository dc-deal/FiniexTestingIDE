# Bar Parity Tests

Cross-pipeline parity tests verify that the simulation pipeline (`execute_tick_loop`) and the AutoTrader pipeline (`AutotraderTickLoop.run()`) produce identical output when fed the same deterministic tick stream.

## What Is Tested

| File | Pair | Tests |
|------|------|-------|
| `tests/parity/test_bar_parity_kraken_spot_btcusd.py` | `(kraken_spot, BTCUSD)` | bar parity, trade parity |
| `tests/parity/test_bar_parity_kraken_spot_ethusd.py` | `(kraken_spot, ETHUSD)` | bar parity, trade parity |

**Current matrix: 4 tests, ~4s.**

Feeds 1000 synthetic BTCUSD ticks through both pipelines and asserts that M1 bar history is identical: bar count, OHLC, volume, tick_count, and timestamp per bar.

## Fixture Design

**1000 synthetic ticks** â€” generated by `make_synthetic_btcusd_ticks()` / `make_synthetic_ethusd_ticks()` in `tests/shared/parity_fixtures.py`. Seeded RNG (seed 42), 1 tick/second, prices bounded Â±10% around the base price. Deterministic: identical output on every call.

**1000 seconds â†’ 16 complete M1 bars** + 1 partial bar at the end. The comparator checks completed bars only (`get_bar_history()` excludes the current incomplete bar).

**100 clipped ticks** â€” `flag_clipped_ticks(every_n=10)` marks every 10th tick as `is_clipped=True`. This is required for the #293 regression guard (see below) and does not affect the expected result: bar rendering is upstream of the clipping gate in both pipelines.

Both pipelines receive the same pre-flagged tick list. Workers and decision logic are replaced with `MagicMock` â€” bars-only phase, no trade execution.

## Regression Validation (#293 Guard)

To confirm the test is load-bearing, temporarily move `bar_rendering_controller.process_tick(tick)` below the `if tick.is_clipped: continue` guard in `python/framework/process/process_tick_loop.py`. With 100 ticks flagged as clipped, the simulation pipeline skips them for bar rendering while the AutoTrader pipeline renders all ticks. Bars diverge â€” the test must fail. Revert the change to restore passing state.

## Phase 3 â€” Trade Parity

One trade parity test per pair, in the same file as the bar parity test.

**Fixture:** flat-price ticks (`make_flat_btcusd_ticks` / `make_flat_ethusd_ticks`) â€” constant bid/ask. Eliminates the 1-tick fill-timing asymmetry between pipelines: simulation fills at tick N+1, AutoTrader at tick N. Constant price â†’ same fill price regardless of timing.

**Executors:**
- Simulation: `TradeSimulator` with `MockBrokerAdapter`, zero latency (`min=max=0 ms`)
- AutoTrader: `LiveTradeExecutor` with `MockBrokerAdapter(INSTANT_FILL)`

`MockBrokerAdapter` provides built-in `SymbolSpecification` for BTCUSD and ETHUSD.

**Decision logic:** `BacktestingDeterministic` â€” LONG at tick 200, close at tick 700 (hold 500 ticks). `worker_orchestrator.process_tick.side_effect` forwards to `decision_logic.compute(tick, {})`.

**Assertions per pair:** bar identity (16 flat M1 bars) + trade identity (entry/exit price, close_reason) + portfolio identity (balance, P&L, trade count).
